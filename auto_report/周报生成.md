# 周报自动生成功能说明

> 最后更新：2026-03-13

---

## 概述

两个脚本实现天府市民云周度分析报告的自动化生成：

| 脚本 | 用途 |
|------|------|
| `search_detail_import.py` | 从友盟 API 拉取搜索词历史数据，入库 `search_detail` 表 |
| `weekly_report_generator.py` | 从数据库读取数据，自动生成 Word 格式周报 |

---

## 周期定义

| 名称 | 范围 |
|------|------|
| **本周期** | 上周五 ~ 本周四（共7天） |
| **上周期** | 上上周五 ~ 上周四（共7天） |

脚本在每周五运行（此时上周四的数据已完整入库），自动计算两个周期的起止日期，无需手动指定。

---

## 脚本一：`search_detail_import.py`

### 功能

参照 `resource_total_history.py` 的逐日入库模式，结合 `5100_detail.py` 的参数列表 API，将 **2026年至昨日** 每天的搜索词数据导入 `search_detail` 表。

### 调用接口

`UmengUappEventParamGetValueListRequest`，参数设置：

```python
req.eventName      = 'search_behavior'
req.eventParamName = 'search_content'
req.startDate      = target_date   # 当天
req.endDate        = target_date   # 当天（逐日拉取）
```

> **API 限制**：该接口每次最多返回 1000 条（按当日搜索量降序排列的 TOP 1000 关键词），为友盟服务端硬限制，无法通过代码突破。对于周报仅需 TOP 10 的场景，该限制不影响结果。

### 数据库写入（`search_detail` 表）

| 字段 | 内容 |
|------|------|
| `search_amount` | 该关键词当日搜索次数 |
| `search_name` | 搜索关键词（已 URL 解码） |
| `stat_date` | 数据所属日期 |
| `port` | 端（安卓 / 苹果 / 鸿蒙） |
| `resource_name` | 固定值 `'search_behavior'` |

**增量逻辑**：以 `(stat_date, port, resource_name)` 为粒度检查重复，已入库的日期自动跳过。

C:\Users\TAOYUAN\PycharmProjects\pythonProject2\venv311\Scripts\python.exe search_detail_import.py

## 报告结构

### 一、日活情况

**数据来源**：`platform_daily_metrics`

#### 1. 日活整体数据

| 生成内容 | 计算逻辑 |
|----------|----------|
| 日活均值 | `platform_dau` 均值 + 约1万（社区保障e管家估算） |
| 不含社区保障e管家环比 | `(本周期均值 - 上周期均值) / 上周期均值 × 100%` |
| 主要原因分析 | **留空，人工填写** |

#### 各端日活变化备注（自动生成）

分析 APP（安卓+苹果+鸿蒙）、支付宝小程序、微信小程序、智慧前端四个端口的环比变化，生成规则：

| 情况 | 生成内容 |
|------|----------|
| 全部上升 | 简洁格式：APP均值 + 支付宝均值 |
| 部分下降 | 列出**所有下降端**：均值 + 环比降幅 + 日均减少量 |
| 全部下降 | 只列**日均减少量最多的两个端**，前缀"各端日活均有所下降" |

示例（全部下降时）：
> 本周期各端日活均有所下降，APP（安卓+苹果+鸿蒙）日活均值16.07万，环比下降7.23%，日均减少1.25万；支付宝小程序日活均值5.30万，环比下降18.56%，日均减少1.21万。

#### 2. 日活本周期趋势

| 生成内容 | 逻辑 |
|----------|------|
| 走势描述文字 | 自动判断"震荡波动/先降后升/先升后降/整体上升/整体下降"，标注峰值日和谷值日（DAU含+1万社治e管家） |
| 折线对比图 | 本周期（实线蓝色）与上周期（虚线橙色）两条线，本周期各日数值标注在线上方 |

**走势判断逻辑**：统计每日方向变化次数，≥2次视为震荡；否则按峰谷位置判断先升后降/先降后升/单调趋势。

---

### 二、服务人次

**数据来源**：`5100_detail`（`510100_items` 事件，`service_name` 参数）

| 生成内容 | 计算逻辑 |
|----------|----------|
| 日均服务人次 | 本/上周期每日 `SUM(service_amount)` 的均值 |
| 日均环比 | `(本周期日均 - 上周期日均) / 上周期日均 × 100%` |
| 涨幅前三 | 本周或上周使用人次 ≥ 1000 的服务，按 `(本-上)/上` 降序取前3 |
| 跌幅前三 | 同上，按升序（最大跌幅）取前3 |
| 峰值日 | 各服务在本/上周期内日汇总最高的日期及次数 |

> 涨跌榜筛选条件：**本周期或上周期总使用人次 ≥ 1000**。

---

### 三、本周总结

| 条目 | 处理方式 |
|------|----------|
| 1）日活趋势 | 自动复用第一部分日活整体数据文字 |
| 2）人均行为次数 | **留空，人工填写** |
| 3）本周宣推事件 | **留空，人工填写** |
| 4）搜索词变化 | 自动从 `search_detail` 生成4列对比表格 |

**搜索词对比表格**（`search_detail` 字段：`search_name` / `search_amount`）：

| 列 | 内容 |
|----|------|
| 增量前十服务名称 | 本周期搜索量比上周期增量最大的前10个关键词 |
| 搜索次数增长量 | 对应增量（本周期总量 - 上周期总量） |
| 本周前十服务名称 | 本周期搜索总量最高的前10个关键词 |
| 搜索次数 | 对应本周期总搜索量 |

> 如 `search_detail` 表为空，显示提示文字，需先运行 `search_detail_import.py`。

## 标准工作流

```
每周五上午
    │
    ├─① 运行 search_detail_import.py   ← 补充昨日（本周四）搜索词数据
    │       （如已入库可跳过，脚本自动去重）
    │
    └─② 运行 weekly_report_generator.py ← 生成本周报 .docx
            │
            └─ 打开生成的 Word 文件，人工填写：
                  · 日活分析原因
                  · 人均行为次数
                  · 本周宣推事件
                  · 搜索词变化分析
```
